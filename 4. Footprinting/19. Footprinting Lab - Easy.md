# Footprinting Lab – DNS/FTP/SSH Enumeration

**Target:** `10.129.42.195` **Date:** June 09, 2026 **Tester:** Valentina

---

## 1. Enumeration

### 1.1 Port Scanning

A full TCP port scan was performed to identify all running services:

```bash
nmap -p- -sV 10.129.42.195
```

**Key findings:**

|Port|Service|Detail|
|---|---|---|
|21/tcp|FTP|ProFTPD – `ftp.int.inlanefreight.htb`|
|22/tcp|SSH|OpenSSH 8.2p1 Ubuntu|
|53/tcp|DNS|ISC BIND 9.16.1|
|2121/tcp|FTP|ProFTPD – **Ceil's FTP**|

The service fingerprint on port 21 revealed the internal domain `ftp.int.inlanefreight.htb`. Port 2121 banner read **"Ceil's FTP"**, which correlated directly with the credentials provided: `ceil:qwer1234`.

---

### 1.2 DNS Enumeration

The target was identified as an internal DNS server. A zone transfer was attempted against `inlanefreight.htb`:

```bash
dig NS inlanefreight.htb @10.129.42.195
dig axfr inlanefreight.htb @10.129.42.195
```

The zone transfer succeeded, exposing the following records:

|Hostname|Type|Value|
|---|---|---|
|inlanefreight.htb|NS|ns.inlanefreight.htb|
|app.inlanefreight.htb|A|10.129.18.15|
|internal.inlanefreight.htb|A|10.129.1.6|
|mail1.inlanefreight.htb|A|10.129.18.201|
|ns.inlanefreight.htb|A|10.129.34.136|

Zone transfers on `internal.inlanefreight.htb` and `int.inlanefreight.htb` failed, indicating restricted zone transfer policies on those subdomains.

## 2. FTP Access – Credential Login

Using the provided credentials, a connection was established to the non-standard FTP port:

```bash
ftp 10.129.42.195 2121
# Username: ceil
# Password: qwer1234
```

After logging in, hidden files were listed using:

```bash
ls -la
```

A `.ssh` directory was found in the user's home directory. Navigating into it revealed:

```
authorized_keys
id_rsa
id_rsa.pub
```

The private key was downloaded:

```bash
cd .ssh
get id_rsa
```

---

## 3. SSH Access via Private Key

The downloaded private key was used to authenticate as `ceil` over SSH:

```bash
chmod 600 id_rsa
ssh -i id_rsa ceil@10.129.42.195
```

Access was granted successfully. The flag was located and retrieved:

```bash
cd flag
cat flag.txt
```

## 4. Vulnerability Summary

|Vulnerability|Description|
|---|---|
|DNS Zone Transfer allowed|Full zone transfer on `inlanefreight.htb` exposes internal hostnames and IPs|
|Weak/shared credentials|`ceil:qwer1234` provided access to a user's personal FTP server|
|SSH private key exposed via FTP|User's `id_rsa` was accessible and readable through FTP, enabling key-based SSH login|
|Non-standard FTP port|Service running on port 2121 may evade basic perimeter controls|

---

## 5. Mitigations

- **DNS:** Restrict zone transfers to authorized secondary nameservers only using ACLs in BIND (`allow-transfer`).
- **FTP credentials:** Enforce strong password policies; avoid reusing credentials across services.
- **SSH keys:** Private keys should never be stored in FTP-accessible directories. Restrict `.ssh` directory permissions to the owner only (`chmod 700 ~/.ssh`).
- **FTP configuration:** Disable directory listings of sensitive home directory contents, or replace FTP with SFTP entirely.

---

## 6. Lessons Learned

- Service banners contain valuable information — the FTP banner directly revealed the username of the account owner, correlating with provided credentials.
- When DNS enumeration stalls, pivot to other open services rather than exhausting all DNS avenues.
- Hidden files (`ls -la`) in FTP sessions frequently contain sensitive material such as shell history, credentials, and SSH keys.
- A successful zone transfer can map out the internal network topology without touching any host directly.
