## Objective

Enumerate the MSSQL service on the target to identify the server hostname and discover non-default databases using valid credentials.

---

## Phase 1: Service Enumeration with Nmap

**Goal:** Identify the MSSQL version and gather service information.

```bash
sudo nmap --script ms-sql-info,ms-sql-empty-password,ms-sql-xp-cmdshell,ms-sql-config,\
ms-sql-ntlm-info,ms-sql-tables,ms-sql-hasdbaccess,ms-sql-dac,ms-sql-dump-hashes \
--script-args mssql.instance-port=1433,mssql.username=sa,mssql.password=,\
mssql.instance-name=MSSQLSERVER -sV -p 1433 10.129.230.249
```

**Why:** Port `1433` is the default MSSQL port. The collection of `ms-sql-*` scripts attempts version detection, credential testing, config enumeration, hash dumping, and more in a single scan.

**Result:** Service confirmed as `Microsoft SQL Server 2019` but all scripts returned errors — the server was likely blocking or restricting script-based queries. Nmap alone wasn't enough here.

---

## Phase 2: Hostname Discovery with Metasploit

**Goal:** Use Metasploit's MSSQL ping module to extract server metadata.

```bash
msfconsole
use auxiliary/scanner/mssql/mssql_ping
set rhosts 10.129.230.249
set verbose true
run
```

**Why:** The `mssql_ping` module queries the SQL Server Browser service (UDP 1434) which responds with instance metadata — without requiring authentication. Setting `verbose true` was critical as the default output returned nothing useful.

**Result:**

```
ServerName    = ILF-SQL-01
InstanceName  = MSSQLSERVER
Version       = 15.0.2000.5
tcp           = 1433
np            = \\ILF-SQL-01\pipe\sql\query
IsClustered   = No
```

**Answer:** `ILF-SQL-01`

Notable findings:

- Named pipes are enabled (`np`) — a potential lateral movement vector
- Not clustered — single instance target

## Phase 3: Database Enumeration with mssqlclient.py

**Goal:** Log in with discovered credentials and enumerate databases.

```bash
python3 mssqlclient.py backdoor@10.129.230.249 -windows-auth
```

**Why:** `mssqlclient.py` from Impacket supports Windows Authentication (`-windows-auth`), which is the default auth mode for MSSQL. This is important — standard SQL auth would fail on a server configured for Windows Auth only. The tool automatically negotiates TLS encryption on connect.

After authenticating:

```sql
select name from sys.databases
```

**Why `sys.databases`:** This system view lists all databases on the instance, including both default system databases and any custom ones created by administrators.

**Result:**

```
master      ← system
tempdb      ← system
model       ← system
msdb        ← system
Employees   ← non-default, custom database
```

**Answer:** `Employees`

---

## Lessons Learned

1. **Nmap scripts aren't always enough** — all ms-sql scripts failed here. Always have a fallback tool (Metasploit, impacket) ready for the same service.
2. **`verbose true` in Metasploit matters** — the mssql_ping module returned nothing without it. Always enable verbose when results seem empty.
3. **Windows Authentication is the MSSQL default** — always try `-windows-auth` first with impacket tools; SQL auth is a secondary option.
4. **Named pipes are a red flag** — `\\ILF-SQL-01\pipe\sql\query` being exposed is a misconfiguration that can enable lateral movement attacks.
5. **`sys.databases` is the first query to run** — it immediately shows what's valuable (custom DBs like `Employees`) vs system noise (`master`, `tempdb`, `model`, `msdb`).
